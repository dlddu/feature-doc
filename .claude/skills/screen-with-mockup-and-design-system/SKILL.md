# Screen with Mockup and Design System

이 레포에서 **새 화면을 코드로 구현할 때**의 진입점 스킬이다.
화면 구현은 두 가지 입력을 가진다: **mockup**(이 화면이 어떻게 보여야 하는가)과 **디자인 시스템**(어떤 시각 언어로 그려야 하는가).
이 스킬은 그 두 입력을 빼먹지 않게 한다.

이 스킬은 mockup이 어떤 형식(HTML, 이미지, Figma 링크 등)인지, 인덱스가 어떤 파일명/필드/구조를 가졌는지 가정하지 않는다.
**이 레포가 mockup을 어떻게 보관하든, 그 방식대로 따라간다.**

---

## 핵심 원칙

1. **mockup 먼저 본다**: 화면을 짜기 전에 이 레포의 mockup 인덱스에서 해당 화면의 mockup이 있는지 확인한다.
2. **없으면 강요하지 않는다**: mockup이 없을 때는 "mockup 없이 진행할까요?"라고 사용자에게 **확인만** 한다.
3. **디자인 시스템은 항상 따른다**: mockup이 있든 없든, 디자인 시스템은 반드시 따른다. UI 작성 부분은 자매 스킬 `ui-with-design-system`의 규율을 그대로 적용한다.
4. **인덱스 갱신 의무는 없다**: 새 mockup 자체를 만드는 건 다른 도구의 영역이다.

---

## 트리거되면 가장 먼저 할 일

새 화면 요청이 들어오면, **코드를 한 줄도 쓰기 전에** 다음을 수행한다.

```
1. 화면 식별
   - 사용자 요청에서 화면 이름/경로/역할을 추출. 예: "회원가입 화면"
2. 이 레포의 mockup 인덱스 위치 파악
   - 일반적인 후보: docs/mockups/, mockups/, design/screens/, README가 가리키는 위치
   - 인덱스 자체가 없으면: "이 레포에 mockup 인덱스가 없습니다.
     mockup 없이 디자인 시스템만 따라 진행할까요?
     아니면 mockup 인덱스를 먼저 만들까요?"
3. 인덱스 안에서 해당 화면의 mockup 항목 검색
4. 디자인 시스템 정의 위치 확인 (없으면 자매 스킬과 같은 안내)
```

---

## 분기: mockup 매칭 결과별 흐름

### 분기 1 — mockup이 있다 (✅ 정상 흐름)

인덱스에 해당 항목이 있고, 참조하는 mockup 파일도 실제로 존재하는 경우.

1. **mockup 파일을 직접 읽는다** (그것이 HTML이든, 이미지든, 링크 대상이든).
2. 인덱스 항목에 디자인 시스템 사용 정보가 함께 적혀 있다면 함께 본다.
3. mockup이 보여주는 화면을 그대로 코드로 옮기되, **모든 시각 표현은 디자인 시스템 정의로 매핑**한다.
   mockup에 임의 값(hex/px 등)이 있더라도 코드에는 디자인 시스템 항목으로 옮긴다.
4. 자매 스킬 `ui-with-design-system`의 매핑 절차를 그대로 따른다.
5. 구현 완료 후 사용자에게 "mockup의 어떤 부분을 어떻게 매핑했는지" 짧게 보고한다.

### 분기 2 — mockup이 인덱스에는 있는데 파일이 없다 (🟡 부분 누락)

인덱스에는 등록돼 있지만 실제 mockup 파일이 없는 경우.

사용자에게 묻는다:
> "이 화면의 mockup이 인덱스에는 있는데 실제 파일은 없습니다.
> (1) 인덱스 정보만 보고 진행할지,
> (2) mockup 없이 진행할지,
> (3) 잠시 멈출지 선택해주세요."

(1)이면 인덱스에 적힌 정보를 가이드로 삼아 디자인 시스템 위에서 구현한다.
(2)면 분기 3으로 가서 진행한다.
(3)이면 멈추고 사용자가 mockup을 추가할 수 있게 기다린다.

### 분기 3 — mockup이 없다 (⚠️ 확인 후 진행)

인덱스에 항목이 없거나 인덱스 자체가 없는 경우.

**다음 메시지로 사용자에게 확인만 한다 (강요하지 않는다):**

> "이 화면의 mockup이 아직 없습니다.
> 어떻게 할까요?
> - **(a) mockup 없이 진행**: 디자인 시스템과 화면 요구사항만으로 구현합니다.
> - **(b) 잠시 멈춤**: 사용자가 별도로 mockup을 마련한 뒤 다시 시작합니다.
> - **(c) mockup 먼저 만들기**: 적절한 도구로 mockup을 먼저 만든 뒤 그것을 참조해 구현합니다."

선택에 따라 진행:

- **(a) 진행**: 사용자에게서 화면 요구사항을 짧게 받는다 (어떤 데이터, 어떤 액션, 어떤 상태들). 그 후 디자인 시스템 위에서 구현. 구현이 끝난 뒤 사용자에게 알린다:
  > "mockup 없이 구현했습니다. 나중에 mockup이 추가되면 인덱스에 등록 권장합니다.
  > 디자인 문서 구조 검증에서 '시각화 누락 단계'로 잡힐 수 있습니다."
- **(b) 멈춤**: 어떤 mockup이 필요한지 한 줄로 정리해 사용자에게 전달하고 종료.
- **(c) mockup 먼저 만들기**: 사용 가능한 mockup 생성 도구를 호출해 mockup을 만든 뒤 분기 1로 돌아간다. 적절한 도구가 없으면 사용자에게 그 사실을 알리고 (a) 또는 (b)로 다시 선택받는다.

## 화면 매칭 휴리스틱

인덱스에서 화면을 찾을 때 다음 우선순위로 본다.

1. **이름 매칭**: mockup 항목의 이름이 사용자 요청의 화면 이름/역할과 명확히 일치.
2. **메타데이터 매칭**: mockup 항목에 적힌 역할/여정/페이지 정보가 사용자 요청과 일치.
3. **유사도 매칭**: 부분 일치. 모호하면 사용자에게 후보를 보여주고 선택받는다.

매칭 후보가 둘 이상이면 자동으로 결정하지 않고 사용자에게 묻는다.

---

## 디자인 시스템 적용 (자매 스킬과의 협업)

화면 구현은 결국 UI 코드 작성이므로, **자매 스킬 `ui-with-design-system`의 규율을 그대로 따른다**:

- 임의 hex/px 값 직접 사용 금지
- 모든 시각 결정은 디자인 시스템 정의로 매핑
- 시스템에 없는 것이 필요하면 시스템 먼저 갱신 후 코드 작성

이 스킬과 자매 스킬은 함께 동작한다. 화면을 새로 짤 때는 두 스킬이 동시 활성화된다고 생각하면 된다.

---

## 구현 후 보고

화면 구현이 끝나면 사용자에게 다음 정보를 알린다. 형식은 이 레포의 다른 보고와 일관되게 유지하면 된다.

- 어떤 mockup을 참조했는지 (없으면 "mockup 없이 구현")
- 어떤 디자인 시스템 항목을 사용했는지
- 디자인 시스템에 새로 추가한 항목이 있는지
- 인덱스 갱신이 필요한지
- 의도적으로 시스템을 우회한 부분이 있는지

---

## 다른 스킬과의 관계

- **ui-with-design-system**: 자매 스킬. 화면을 짜는 동안 UI 코드 작성 부분은 그 스킬의 규율을 따른다.

## 실패 모드와 회피

| 실패 모드 | 증상 | 회피 |
|-----------|------|------|
| mockup 없는데 그냥 코드부터 씀 | 화면 요구사항 빠짐, 사용자 의도와 다른 결과 | 트리거 시 mockup 매칭 분기 먼저 |
| mockup이 있는데 안 보고 짐 | mockup과 결과가 다름 | 인덱스 매칭을 항상 먼저 시도 |
| mockup 없을 때 무리하게 강요 | 사용자가 도구 없는 환경에서 막힘 | 강요하지 않고 (a)/(b)/(c) 선택지 제시 |
| mockup의 임의 hex를 코드에 그대로 옮김 | 디자인 시스템 무력화 | 자매 스킬의 매핑 절차로 토큰화 |
| 인덱스에 없는 mockup 발견했는데 무시 | 인덱스 거짓말 누적 | 사용자에게 "이걸 인덱스에 등록할까요?" 안내 |
