# UI with Design System

이 레포의 UI 코드는 이 레포에 정의된 디자인 시스템 안에서만 산다.
이 스킬의 역할은 그 규율을 강제하고, 시스템에서 벗어나려는 모든 시도를 사용자가 의식적으로 결정하게 만드는 것이다.

이 스킬은 디자인 시스템이 어떤 폴더에, 어떤 파일명으로, 어떤 명명 규칙/메타데이터 형식으로 적혀 있는지 가정하지 않는다.
**이 레포가 사용하는 위치·구조·규칙을 그대로 따른다.**

---

## 핵심 원칙

1. **임의 값 금지**: hex 색상, 임의 px/rem 값, 인라인 스타일을 직접 적지 않는다. 모두 디자인 시스템에 정의된 토큰을 거친다.
2. **시스템 우선**: 새 컴포넌트가 필요해 보여도 먼저 디자인 시스템에 같은 역할의 것이 있는지 본다.
3. **시스템 갱신은 명시적**: 시스템에 없는 항목이 필요하면, **코드에 먼저 박지 않고** 디자인 시스템 정의에 먼저 추가한 뒤 코드에서 참조한다.
4. **사용자 결정**: 시스템을 벗어나야 할 정당한 이유가 있다면 그건 사용자가 결정한다. 임의로 빠져나가지 않는다.

---

## 트리거되면 가장 먼저 할 일

UI 관련 작업이 들어오면, **코드를 한 줄도 쓰기 전에** 다음을 수행한다.

```
1. 이 레포의 디자인 시스템 정의 위치 파악
   - 일반적인 후보: docs/design-system/, design/, design-tokens/, README나 CONTRIBUTING이 가리키는 위치
   - 어디에도 없으면: "이 레포에 디자인 시스템이 정의돼 있지 않습니다.
     디자인 시스템을 먼저 만들까요, 아니면 시스템 없이 진행할까요?"
     라고 사용자에게 묻고 결정 대기.
2. 디자인 시스템 정의를 모두 읽는다
   - 토큰/컴포넌트/패턴이 별도 파일이든 한 파일이든, 정의된 모든 것을 본다.
3. 사용자가 요청한 UI 작업을 그 정의에 매핑한다 (다음 절 참조).
```

이 단계를 건너뛰고 코드부터 작성하는 것은 이 스킬의 가장 큰 실패 모드다.

---

## 작업 매핑 절차

요청을 받으면 다음 4단계로 매핑한다.

### 1. 필요한 시각 요소 식별

요청에 등장하는 시각적 결정을 모두 나열한다. 색상, 폰트, 간격, 모서리, 그림자, 컴포넌트 종류, 변형, 상태 등.

### 2. 시스템과 매칭

각 결정을 디자인 시스템 정의와 매칭한다. 매칭되는 것과 안 되는 것을 구분한다.

### 3. 시스템에 없는 것 처리

매칭이 안 되는 항목이 있으면 **사용자에게 분기를 보여준다**. 절대 임의로 코드에 박지 않는다.

선택지는 셋:
- **(A) 시스템에 추가**: 디자인 시스템 정의를 먼저 갱신한 뒤 코드에 반영. 권장 기본값.
- **(B) 기존 항목으로 대체**: 가까운 기존 항목으로 처리할 수 있는지 본다.
- **(C) 일회성 예외 수용**: 코드에 임의 값 들어가는 걸 사용자가 명시적으로 수용. 이 경우 코드에 `// design-system-exception: <사유>` 주석을 단다.

### 4. 구현

매핑이 완료된 후에만 코드를 작성한다. 작성 후:
- 디자인 시스템 정의를 갱신한 경우, 그 사실을 사용자에게 알린다.
- 새로 도입한 항목이 있으면 어디에 어떻게 추가했는지 보고한다.

## 자주 만나는 패턴

### "그냥 색만 좀 바꿔줘"

색을 받으면 먼저 디자인 시스템에서 가까운 의미적 토큰을 찾는다.
- 적당한 토큰이 있으면 그 토큰을 적용한다.
- 없으면 사용자에게 묻는다: "이 색을 어떤 의미적 이름으로 시스템에 추가할까요?"

### "이 컴포넌트 좀 만들어줘"

새 컴포넌트 요청이면:
1. 디자인 시스템에서 같은 역할의 컴포넌트가 있는지 본다 (이름이 달라도 역할로 검색).
2. 있으면 그것의 변형으로 처리할 수 있는지 검토.
3. 정말 새 컴포넌트가 맞으면 디자인 시스템에 정의를 먼저 추가하고, 코드를 그 정의에 맞게 작성.

### "Tailwind 클래스 좀 정리해줘"

사용 중인 임의 값(예: `text-[#3B82F6]`, `p-[13px]` 같은 arbitrary values)을 토큰으로 치환할 수 있는지 본다.
프로젝트의 스타일 설정이 디자인 시스템과 동기화돼 있는지도 확인하고, 안 돼 있으면 사용자에게 알린다.

### "디자인 시스템 따로 없이 그냥 빠르게"

이 요청이 오면 멈추고 사용자에게 확실히 한다:
> "디자인 시스템을 일시적으로 우회하시겠습니까? 우회하면 일관성이 깨질 수 있고,
> 나중에 디자인 문서 구조 검증에서 '임의 스타일'로 잡힐 수 있습니다.
> 그래도 진행하시겠어요?"

사용자가 동의하면 진행한다. 단, 결과 코드에 한 곳 이상 `// design-system-bypass: <사유>` 주석을 단다.

## 시스템 갱신 시 주의

이 스킬이 디자인 시스템 정의를 직접 수정할 수 있다. 다만 다음 규칙을 따른다.

- **추가만**: 기존 항목을 무단 수정/삭제하지 않는다. 변경이 필요하면 사용자에게 묻는다.
- **이 레포의 규칙 따르기**: 이 레포가 사용하는 명명 규칙·구조·메타데이터 형식을 그대로 따른다. 새로운 규칙을 임의로 도입하지 않는다.

## 실패 모드와 회피

| 실패 모드 | 증상 | 회피 |
|-----------|------|------|
| 디자인 시스템 안 읽고 코드부터 씀 | 임의 hex 값이 코드에 박힘 | 트리거 시 무조건 시스템 정의 먼저 읽기 |
| "비슷한 거 있을 것 같아서" 검색 안 함 | 중복 컴포넌트 생성 | 시스템을 역할 기준으로 검색 |
| 시스템에 없는 항목을 코드에 박고 끝 | 시스템과 코드의 균열 | 코드 박기 전에 시스템 먼저 갱신 |
| 사용자가 "빨리"라고 해서 우회 | 일관성 붕괴 | 우회 시 사용자에게 명시적 확인 + 주석 |
