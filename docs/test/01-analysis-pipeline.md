# 테스트 문서: 코드베이스 분석 파이프라인

## 검증 대상 AC
- AC1.1: 저장소 연결 및 분석 트리거 (PRD: 코드베이스 분석 파이프라인)
- AC1.2: 횡단 관심사 자동 추출 및 문서화 (PRD: 코드베이스 분석 파이프라인)
- AC1.3: feature 탐색 전략 생성 (PRD: 코드베이스 분석 파이프라인)
- AC1.4: feature 후보 목록 추출 (PRD: 코드베이스 분석 파이프라인)
- AC1.5: 분석 작업의 비동기 실행과 진행 상황 가시성 (PRD: 코드베이스 분석 파이프라인)

## 테스트 시나리오

### 시나리오 1: 정상 저장소 연결 및 전체 파이프라인 실행
- **사전 조건**: FeatureDoc GitHub App 설치 완료, LLM API Key 등록 완료, 접근 가능한 공개 저장소 URL 보유.
- **실행 단계**:
  1. 모바일에서 저장소 URL과 브랜치 입력
  2. "분석 시작" 탭
  3. 횡단 분석 단계 완료 대기
  4. 자동 생성된 탐색 전략 검토 후 승인
  5. feature 후보 목록 확인
- **기대 결과**: 횡단 관심사 문서, 탐색 전략, feature 후보 목록 순서로 산출물이 단계별로 제시되며, 각 산출물에 코드 근거가 첨부된다.
- **검증 AC**: AC1.1, AC1.2, AC1.3, AC1.4

### 시나리오 2: 접근 범위 밖 저장소 접근 시도
- **사전 조건**: GitHub App 설치 시 접근을 허용하지 않은 저장소 URL.
- **실행 단계**: 저장소 URL 입력 → 분석 시작
- **기대 결과**: "이 저장소에 접근할 수 없습니다 / App 설치 범위를 확인해 주세요"와 같은 명확한 메시지가 표시되고, App 설치 관리 화면으로 이동 가능하다. 분석 작업은 큐에 등록되지 않는다.
- **검증 AC**: AC1.1

### 시나리오 3: 횡단 분석 결과의 결정적 재현
- **사전 조건**: 시나리오 1을 1회 완료한 저장소.
- **실행 단계**: 동일 커밋 SHA로 재분석 트리거
- **기대 결과**: 횡단 관심사 항목과 그 근거 파일 목록이 이전 결과와 동일하거나, 차이가 있다면 차이 사유(LLM 응답 비결정성 등)가 명시된다.
- **검증 AC**: AC1.2

### 시나리오 4: 탐색 전략의 사용자 편집 후 재사용
- **사전 조건**: 횡단 분석 완료, 자동 생성된 탐색 전략 존재.
- **실행 단계**: 전략에서 한 항목을 추가/삭제 → 승인 → feature 추출 단계 진행
- **기대 결과**: 사용자 편집이 반영된 전략으로 추출이 수행되고, 결과 feature 목록이 편집 전과 달라진다.
- **검증 AC**: AC1.3, AC1.4

### 시나리오 5: 분석 중 앱 종료 후 복귀
- **사전 조건**: 분석이 횡단 분석 단계에서 진행 중.
- **실행 단계**: 모바일 앱 강제 종료 → 1분 후 재시작 → 분석 화면 진입
- **기대 결과**: 진행률과 현재 단계가 그대로 표시되며, 백그라운드에서 진행된 작업 결과가 반영되어 있다. 사용자는 같은 화면에서 이어서 작업할 수 있다.
- **검증 AC**: AC1.5

### 시나리오 6: 특정 단계 실패 후 부분 재시도
- **사전 조건**: feature 추출 단계가 LLM 호출 한도 초과로 실패한 상태.
- **실행 단계**: 실패 화면에서 "이 단계만 다시 시도" 탭
- **기대 결과**: 횡단 분석/탐색 전략은 그대로 유지되며, feature 추출 단계만 재실행된다.
- **검증 AC**: AC1.5

### 시나리오 7: 추출된 feature 후보의 거부와 재분석 시 회피
- **사전 조건**: feature 후보가 5개 추출됨.
- **실행 단계**: 1번 후보를 "feature 아님" 사유와 함께 거부 → 같은 저장소 재분석
- **기대 결과**: 재분석 시 1번 후보가 동일한 위치에서 다시 발견되더라도, 사용자에게 "이전에 거부된 항목"으로 표시된다.
- **검증 AC**: AC1.4
