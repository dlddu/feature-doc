# 테스트 문서: 플랫폼 (k8s · 자격증명 · 모바일 우선)

## 검증 대상 AC
- AC4.1: 사용자 GitHub Token 등록·교체·폐기 (PRD: 플랫폼)
- AC4.2: 사용자 LLM API Key 등록·교체·폐기 및 호출 위임 (PRD: 플랫폼)
- AC4.3: 자격증명의 안전한 사용 정책 (PRD: 플랫폼)
- AC4.4: 모바일 우선 UI/UX (PRD: 플랫폼)
- AC4.5: k8s 배포 및 워크로드 분리 (PRD: 플랫폼)
- AC4.6: 관측 가능성과 비용 가시성 (PRD: 플랫폼)
- AC4.7: 분석 작업의 격리 (PRD: 플랫폼)

## 테스트 시나리오

### 시나리오 1: GitHub Token 등록과 권한 안내
- **사전 조건**: 토큰 없음.
- **실행 단계**: 토큰 등록 화면 진입 → 안내된 권한 범위 확인 → 토큰 입력 → 저장
- **기대 결과**: 등록 전에 필요한 최소 scope 안내가 표시된다. 등록 후 토큰 평문은 어디에서도 다시 표시되지 않는다.
- **검증 AC**: AC4.1, AC4.3

### 시나리오 2: 진행 중 분석 도중 토큰 폐기
- **사전 조건**: 분석 1건이 진행 중.
- **실행 단계**: 토큰 폐기 → 진행 상황 확인
- **기대 결과**: 사전 명시된 정책에 따라 분석이 중단되거나 현재 호출까지만 마무리된다. 그 결과가 사용자에게 통지된다.
- **검증 AC**: AC4.1

### 시나리오 3: LLM API Key 등록 후 호출 위임
- **사전 조건**: 키 없음.
- **실행 단계**: 키 등록 → 임의 분석 트리거 → LLM 제공자 측 호출 로그 확인
- **기대 결과**: 모든 LLM 호출이 사용자 키로 발생한다. 시스템 측 키는 사용되지 않는다.
- **검증 AC**: AC4.2

### 시나리오 4: 키 폐기 후 신규 호출 차단
- **사전 조건**: 키 등록 상태에서 분석 진행 중.
- **실행 단계**: 키 폐기 → 새 분석 트리거 시도
- **기대 결과**: 신규 LLM 호출이 차단되며, "키가 없거나 폐기되었습니다" 메시지가 표시된다.
- **검증 AC**: AC4.2

### 시나리오 5: 자격증명의 로그 노출 검증
- **사전 조건**: 토큰/키가 등록된 상태에서 정상 분석 1회 + 의도적 오류 1회 유발.
- **실행 단계**: 시스템 로그/에러 리포트 검사
- **기대 결과**: 어떤 로그/오류 메시지에서도 토큰·키의 평문이나 그 일부가 노출되지 않는다.
- **검증 AC**: AC4.3

### 시나리오 6: 모바일 우선 UI 검증 (5분 세션)
- **사전 조건**: 분석이 완료되어 feature 목록과 인수 테스트가 존재.
- **실행 단계**: 모바일 폭(예: 390px) 환경에서 (a) 분석 진행 확인, (b) feature 1개 검토, (c) LLM 보조 수정 1회, (d) 의존성 1건 조회 — 5분 안에 수행
- **기대 결과**: 모든 작업이 한 손 조작과 5분 내에 완료 가능. 긴 결과는 요약 → 상세 단계 노출이 적용된다.
- **검증 AC**: AC4.4

### 시나리오 7: 워커 다운 시 API 가용성
- **사전 조건**: k8s에 API와 워커 워크로드가 분리되어 배포된 상태.
- **실행 단계**: 워커 파드 모두 강제 종료 → API에 대한 사용자 요청(자격증명 조회, 결과 조회) 시도
- **기대 결과**: API는 정상 응답하며, 신규 분석 요청은 큐에 적재되어 워커 복구 후 처리된다. 사용자에게 "분석 작업이 대기 중입니다" 표시.
- **검증 AC**: AC4.5

### 시나리오 8: 워커의 수평 확장
- **사전 조건**: 다수 분석 요청이 큐에 적재된 상태.
- **실행 단계**: 워커 replica를 늘림
- **기대 결과**: 처리량이 비례적으로 증가하고, API/워커 간 결합 없이 확장된다.
- **검증 AC**: AC4.5

### 시나리오 9: 사용자별 비용 가시성
- **사전 조건**: 한 사용자가 여러 분석을 수행.
- **실행 단계**: 사용자 화면에서 누적 LLM 호출 횟수와 토큰 사용량 확인
- **기대 결과**: 작업별·전체별 호출 횟수와 추정 비용이 표시된다. 운영자 화면에서도 동일한 데이터에 접근 가능.
- **검증 AC**: AC4.6

### 시나리오 10: 사용자 간 데이터 격리
- **사전 조건**: 사용자 A, B가 각각 다른 저장소를 분석한 상태.
- **실행 단계**: 사용자 A 인증으로 사용자 B의 자원(자격증명/저장소/feature 데이터)에 직접 API 호출 시도
- **기대 결과**: 모든 시도가 권한 거부로 차단된다. 워커 작업 컨텍스트도 사용자별로 격리되어 동일 워커 인스턴스에서 처리되어도 데이터가 섞이지 않는다.
- **검증 AC**: AC4.7
