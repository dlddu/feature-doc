# 테스트 문서: Feature 단위 문서 관리 (LLM 활용 CRUD)

## 검증 대상 AC
- AC3.1: feature 문서의 LLM 보조 수정 (PRD: 문서 관리)
- AC3.2: feature 문서의 추가 (PRD: 문서 관리)
- AC3.3: feature 문서의 삭제와 보존 (PRD: 문서 관리)
- AC3.4: 변경 이력 추적과 출처 보존 (PRD: 문서 관리)
- AC3.5: 코드 자동 분석과 사용자 편집의 충돌 처리 (PRD: 문서 관리)

## 테스트 시나리오

### 시나리오 1: 모바일에서 LLM 보조 수정 3탭 흐름
- **사전 조건**: 인수 테스트 문서가 존재하는 feature.
- **실행 단계**:
  1. feature 선택 → "수정 요청" 입력 ("에러 케이스 1개 더 추가")
  2. LLM 제안 diff 확인
  3. "승인" 탭
- **기대 결과**: 3탭 이내 작업 완료. 적용 후 문서에 새 시나리오가 추가되며 변경 출처가 "사용자 by LLM"으로 기록된다.
- **검증 AC**: AC3.1, AC3.4

### 시나리오 2: 잘못된 LLM 제안의 거부와 학습
- **사전 조건**: 시나리오 1과 동일.
- **실행 단계**: 같은 종류의 수정 요청 → LLM이 부적절한 제안 → 사유와 함께 거부
- **기대 결과**: 거부 사유가 기록된다. 다음 유사 요청 시 거부된 패턴을 회피하려는 시도(프롬프트 컨텍스트 활용)가 일어난다.
- **검증 AC**: AC3.1

### 시나리오 3: 자동 추출되지 않은 feature 수동 추가 (코드 근거 있음)
- **사전 조건**: 코드에는 존재하지만 자동 추출에서 누락된 feature가 있음.
- **실행 단계**: "회원 탈퇴 기능 추가" → LLM이 코드 검색 → 초안 제시 → 사용자 확정
- **기대 결과**: 인수 테스트 초안과 의존성 후보가 코드 근거와 함께 제시된다. 사용자가 확정하면 새 feature로 등록된다.
- **검증 AC**: AC3.2

### 시나리오 4: 코드 근거를 찾을 수 없는 feature 추가 시도
- **사전 조건**: 코드에 존재하지 않는 기능에 대한 추가 요청.
- **실행 단계**: "관리자 일괄 삭제 추가" → LLM이 코드 검색 실패
- **기대 결과**: "코드에서 근거를 찾지 못했습니다"라는 메시지가 표시되며, 강제로 만들어내지 않는다. 사용자는 수동으로 강제 추가하거나 취소할 수 있다.
- **검증 AC**: AC3.2

### 시나리오 5: feature 삭제와 복구
- **사전 조건**: 등록된 feature 1개.
- **실행 단계**: "삭제" 탭 → 사유 입력 → 확인
- **기대 결과**: feature는 보관소로 이동하며, 일정 기간 내 복구 가능. 즉시 영구 삭제되지 않는다.
- **검증 AC**: AC3.3

### 시나리오 6: 삭제된 feature의 자동 재발견 표시
- **사전 조건**: 시나리오 5에서 삭제된 feature.
- **실행 단계**: 같은 저장소 자동 재분석
- **기대 결과**: 동일 feature가 자동으로 발견되더라도 "이전에 거부/삭제된 항목입니다"로 표시되며, 사용자 확인 없이는 다시 활성화되지 않는다.
- **검증 AC**: AC3.3

### 시나리오 7: 자동 분석 결과와 사용자 편집의 충돌
- **사전 조건**: 사용자가 인수 테스트 시나리오 1개를 직접 편집한 feature가 있고, 코드가 변경되어 자동 재분석이 동일 시나리오를 다르게 갱신하려 함.
- **실행 단계**: 재분석 완료 후 충돌 화면 진입 → "병합(LLM 보조)" 선택
- **기대 결과**: 사용자 편집이 자동으로 덮이지 않고, 자동 결과/사용자 편집 양쪽이 표시된다. 병합 선택 시 LLM이 통합 제안을 만들고 사용자 승인을 요구한다.
- **검증 AC**: AC3.5

### 시나리오 8: 변경 이력 조회와 임의 시점 복원
- **사전 조건**: 한 feature에 자동 변경 1회, 사용자 LLM 보조 변경 1회, 사용자 직접 변경 1회 발생.
- **실행 단계**: 이력 화면에서 두 번째 변경 시점 선택 → "이 시점으로 복원"
- **기대 결과**: 각 변경의 출처(자동/사용자 by LLM/사용자 직접)가 명확히 구분되어 보이고, 선택한 시점의 상태로 정확히 복원된다.
- **검증 AC**: AC3.4
