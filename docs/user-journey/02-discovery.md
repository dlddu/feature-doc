# 플로우 2 — 자동 발견 (Discovery)

> 큐에 등록된 분석 작업이 4단계 파이프라인을 돌면서 횡단 관심사·탐색 전략·feature 후보를 차례로 만들고, 사용자가 각 산출물을 검토·승인하는 흐름.
>
> 페르소나·전체 개요는 [`README.md`](./README.md)를 보세요. 이 문서는 **이 플로우 안의 화면 단위 흐름**만 다룹니다.

## 의도와 맥락

**사용자 의도**: "내 코드베이스에서 최종 사용자에게 노출되는 기능들이 뭔지, 무엇 위에서 돌아가고 있는지 한 번에 파악하고 싶다."

이 플로우의 핵심 특징은 **사용자가 능동적으로 머물지 않는다**는 점입니다. 분석은 분 단위로 길고, 사용자는 앱을 닫고 다른 일을 합니다. 따라서 화면은 (a) 백그라운드에서 살아 있는 진행 상황을 보여주고 (b) 단계 완료 시 사용자를 다시 불러오는 두 역할을 동시에 합니다.

**주된 페르소나**: P1 (Pre)만 들어옵니다. P2는 결과를 받아만 보지 이 플로우 안의 결정권은 갖지 않아요.

## 화면 흐름

```
[S04]                 [S05]                  [S06]                  [S07]
분석 진행      →      횡단 관심사       →    탐색 전략       →     feature 후보
(5단계 진행률)        (인프라/아키텍처/      (LLM이 만든            (승인/거부/
                      프레임워크/미들웨어)   전략 검토 + 편집)      병합/이름변경)
```

S04는 **항상 떠 있는 진행 상황**이고, 단계 완료마다 S05 → S06 → S07이 차례로 보여집니다. 한 단계가 실패하면 그 단계만 재시도 가능 (AC1.5).

## 화면별 상세

### S04 — 분석 진행

[`s04-analysis-progress.svg`](../wireframes/s04-analysis-progress.svg)

| 항목 | 내용 |
|------|------|
| **사용자 의도** | "지금 어디까지 갔는지, 얼마나 더 걸릴지, 비용은 얼마나 쓰고 있는지 알고 싶다." |
| **주요 행동** | (1) 5단계 진행률 확인 (2) 누적 LLM 호출/추정 비용 확인 (3) 실패한 단계가 있으면 그 단계만 재시도 (4) 앱을 닫고 떠나기 |
| **시스템 응답** | 단계별 상태(대기/진행/완료/실패) · 누적 비용 갱신 · 알림 권한이 있으면 단계 완료 시 푸시 |
| **이 화면의 끝** | 다음 산출물 단계(S05/S06/S07)가 완료되면 자동으로 그 화면으로 진입 가능, 또는 사용자가 앱을 닫음 |

**이 화면의 약속** (V7, V8 / AC1.5, AC4.6):
- 앱을 닫고 다시 열어도 진행 상태가 그대로 유지된다.
- 단계가 실패해도 전체 파이프라인이 죽지 않는다 — 그 단계만 재시도 가능.
- 누적 비용이 *항상* 보인다 (사용자가 통제감을 잃지 않도록).

**마찰점 (F3)**: 단계 실패 시 처음부터 다시 돌리게 만들면 사용자는 분석 시작 자체에 다시 들이는 심리적 비용이 커집니다. 단계별 재시도가 핵심.

**페르소나 메모**:
- P1: 진행 중 화면을 자주 보지는 않지만, 완료 알림에 의존함. 알림 없이 한 시간 비어 있는 게 가장 큰 불안.

### S05 — 횡단 관심사

[`s05-cross-cutting-concerns.svg`](../wireframes/s05-cross-cutting-concerns.svg)

| 항목 | 내용 |
|------|------|
| **사용자 의도** | "내 코드가 어떤 인프라/아키텍처/프레임워크/미들웨어 위에 서 있는지를 외부의 눈으로 한 번 정리받고 싶다." |
| **주요 행동** | (1) 추출된 항목들을 카테고리별로 훑기 (2) 각 항목의 근거 파일/심볼 확인 (3) 누락이 있으면 인지 (이 단계에서 직접 편집은 없음) |
| **시스템 응답** | 카테고리별로 정리된 횡단 관심사 문서 · 항목마다 코드 근거 첨부 · 동일 repo 재분석 시 결과가 결정적이거나 차이가 명시됨 |
| **이 화면의 끝** | 사용자는 결과를 인지한 상태로 [S06](../wireframes/s06-discovery-strategy.svg)으로 이동 |

**이 화면의 약속** (V2, V4 / AC1.2):
- 모든 추출 항목에 **코드 근거**(파일 경로/심볼)가 붙는다. 근거 없는 항목은 표시되지 않거나 "근거 없음"으로 명시된다.
- 같은 커밋을 두 번 분석했을 때 결과가 같거나, 다르면 *무엇이* 다른지가 명시된다.

**마찰점 (F4)**: LLM 비결정성으로 결과가 흔들리는 게 가장 큰 신뢰 위협. "차이가 있으면 항목별로 명시"의 정책 합의 없이 운영하면 P1은 결과를 점점 덜 믿게 됩니다.

**페르소나 메모**:
- P1: 자기 코드 구조를 외부의 눈으로 처음 보는 순간. 누락/오인이 있으면 즉시 캐치.

### S06 — 탐색 전략 검토

[`s06-discovery-strategy.svg`](../wireframes/s06-discovery-strategy.svg)

| 항목 | 내용 |
|------|------|
| **사용자 의도** | "내 코드에서 'feature'를 어떻게 발견할지의 전략이 그럴듯한지 보고, 내가 아는 부분을 보태고 싶다." |
| **주요 행동** | (1) LLM이 만든 전략(진입점 패턴/라우팅 위치/UI 컴포넌트 경로/API 엔드포인트/CLI 등)을 항목별 검토 (2) 항목 추가/삭제/수정 (3) 승인 |
| **시스템 응답** | 횡단 분석 결과 변경 시 전략 재생성 가능 · 승인된 전략만 다음 단계 입력으로 사용 · 사용자 편집 이력 보존 |
| **이 화면의 끝** | 승인된 전략이 다음 단계(feature 후보 추출)에 입력으로 전달됨 |

**이 화면의 약속** (V1, V2 / AC1.3):
- LLM이 만든 전략은 **승인 전 입력으로 쓰이지 않는다**. 사용자의 명시적 결정이 항상 게이트.
- 사용자 편집은 다음 분석에서 참조될 수 있도록 보존된다.

**결정의 무게**: 이 화면이 최종 feature 목록의 **품질을 결정하는 첫 게이트**입니다. 전략을 그대로 받느냐, 본인이 아는 코드 구조를 보태느냐의 한 줄 차이가 [S07](../wireframes/s07-feature-candidates.svg)의 후보 목록을 크게 바꿉니다.

**페르소나 메모**:
- P1: 자기 코드의 *비표준 진입점*(예: 전용 admin CLI, 특수 라우팅)을 알기 때문에 여기서 한 줄 보태는 게 자연스러움.

### S07 — feature 후보 검토

[`s07-feature-candidates.svg`](../wireframes/s07-feature-candidates.svg)

| 항목 | 내용 |
|------|------|
| **사용자 의도** | "이 후보 중 진짜 사용자에게 노출되는 기능과 그렇지 않은 것을 가르고 싶다." |
| **주요 행동** | (1) 후보별 발견 위치/근거 확인 (2) 승인/거부/병합/이름 변경 (3) 거부에는 사유 기록 |
| **시스템 응답** | 각 후보에 발견 위치(파일·심볼)와 추정 근거 첨부 · 거부 사유는 다음 분석을 위해 보존 · 승인된 후보 = 확정된 feature |
| **이 화면의 끝** | 확정된 feature 목록이 만들어지고, 이 플로우가 종료됨. 사용자는 [플로우 3](./03-feature-documents.md)로 이동 |

**이 화면의 약속** (V1, V2, V4 / AC1.4):
- 모든 후보에 **발견 위치와 근거**가 붙는다.
- 거부 사유는 보존되고, 다음 자동 분석에서 같은 후보가 다시 잡히면 "이전에 거부된 항목"으로 표시됨 ([플로우 4](./04-continuous-maintenance.md)에서 다시 등장).

**가장 책임 있는 결정**: 이 화면은 P1이 가장 신중해야 하는 지점. 잘못 거부한 feature는 다음 분석에서도 거부 표시로 들어오기 때문에, *"왜 거부했는지"*를 본인이 1주일 후에도 이해할 수 있게 적어 둬야 합니다.

**페르소나 메모**:
- P1: 자기 코드라서 디버그 엔드포인트/내부 도구를 빠르게 식별 가능. 사유 기록의 명료함이 다음 재분석의 품질을 좌우.

## 이 플로우의 시나리오

**14:06–14:24 · 카페** — Pre는 분석을 시작하고 앱을 닫음. 18분 뒤 단계 완료 알림이 와서 다시 연다. [S04](../wireframes/s04-analysis-progress.svg)의 진행 상태가 그대로 살아 있음 (AC1.5).

**14:24** — [S05](../wireframes/s05-cross-cutting-concerns.svg). 인프라 · 아키텍처 · 프레임워크 · 미들웨어가 항목별로 정리돼 있음. "Express 4.x · PostgreSQL · Redis 캐시 미들웨어"… 익숙한 자기 코드의 구조가 외부의 눈으로 정리돼 있는 첫 경험. 누락 없음, 그대로 다음으로.

**14:27** — [S06](../wireframes/s06-discovery-strategy.svg). LLM이 만든 전략에 "Express 라우터 정의 위치 · React 페이지 컴포넌트 진입점 · CLI 명령" 3개. Pre는 자기 프로젝트에 admin CLI가 따로 있다는 걸 알기 때문에 한 줄 추가 편집 후 승인.

**14:30** — [S07](../wireframes/s07-feature-candidates.svg). 후보 14개. 그중 2개는 사실 내부 디버그 엔드포인트라서 거부 (사유: "디버그 전용, 사용자 노출 아님"). 나머지 12개를 승인.

이 플로우 종료 시점에 **확정된 feature 12개** 가 다음 플로우의 입력이 됨.

이 플로우 안에서 사용자가 들인 능동 시간: **약 6분** (S05·S06·S07 검토). 분석 자체의 백그라운드 시간은 18분.

## 플로우 종료 조건과 다음 플로우

이 플로우는 **확정된 feature 목록이 만들어진 시점**에 종료됩니다. 그 시점부터:

- **[플로우 3 — Feature 문서](./03-feature-documents.md)** 가 시작 가능. 단, 자동으로 진입하지 않고 사용자가 한 feature를 골라 들어가는 능동 행동이 필요.
- 거부된 후보들은 [플로우 4 — 지속적 유지](./04-continuous-maintenance.md)의 "재발견 시 회피" 흐름의 입력으로 보존됨 (AC1.4).
