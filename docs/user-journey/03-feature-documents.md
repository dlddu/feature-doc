# 플로우 3 — Feature 문서 검토와 활용 (Feature Documents)

> 확정된 feature 1개의 인수 시나리오를 검토하고, 종단 의존성을 보고, 필요하면 LLM에게 자연어로 수정 요청을 보내는 흐름.
>
> 페르소나·전체 개요는 [`README.md`](./README.md)를 보세요. 이 문서는 **이 플로우 안의 화면 단위 흐름**만 다룹니다.

## 의도와 맥락

**사용자 의도**:
- P1: "확정된 feature 1개의 표현이 정확한지 검토하고, 의존성을 한눈에 본다."
- P2: "이 feature가 무엇을 하는지 1~2분 안에 이해하고, 문제가 있으면 자연어 한 줄로 수정 제안을 보낸다."

이 플로우는 **P2가 처음 들어오는 지점**입니다. 비개발자가 봤을 때 의미가 통해야 한다는 검수 기준이 [S08](../wireframes/s08-feature-acceptance.svg)에 가장 강하게 걸립니다.

**주된 페르소나**: P1, P2 둘 다.

## 화면 흐름

```
[S08]                       [S09]                     [S10]
인수 시나리오         →     종단 의존성        →      LLM 보조 편집
(Given-When-Then)           (그래프 + 카테고리)        (자연어 → diff → 승인)
```

화면은 **연속적이지 않고 병렬적**이에요. 사용자는 한 feature 안에서 S08↔S09를 자유롭게 오가며, 필요할 때 S10을 띄웁니다. 흐름의 끝은 "이 feature를 검토 끝났다"고 사용자가 결정하는 시점.

## 화면별 상세

### S08 — 인수 시나리오

[`s08-feature-acceptance.svg`](../wireframes/s08-feature-acceptance.svg)

| 항목 | 내용 |
|------|------|
| **사용자 의도** | "이 feature를 사용자가 썼을 때 어떤 일이 벌어지는지를, 가능한 모든 경우에 대해 알고 싶다." |
| **주요 행동** | (1) 시나리오를 Given-When-Then 형태로 읽기 (2) 각 시나리오의 코드 근거 링크 확인 (3) "코드와 테스트가 다르게 말하는 부분"이 있다면 별도 섹션에서 확인 |
| **시스템 응답** | 시나리오 단위 구조화 · 비개발자 언어 사용 · 각 시나리오에 코드 근거 첨부 · 모순(로직 vs 테스트) 별도 섹션 |
| **이 화면의 끝** | 사용자는 시나리오 이해를 마치고 [S09](../wireframes/s09-feature-dependencies.svg)로 의존성을 보거나 [S10](../wireframes/s10-llm-edit.svg)로 편집 요청 |

**이 화면의 약속** (V3, V4 / AC2.1, AC2.2, AC2.3):
- 시나리오는 **개발자 용어로 적히지 않는다**. 비개발자(P2)가 읽고 그 feature가 무엇을 하는지 설명할 수 있어야 함.
- 각 시나리오는 *로직 코드*에서 도출되고, *테스트 코드*로 보강된다.
- 로직과 테스트가 모순되는 경우 별도 섹션 ("코드와 테스트가 다르게 말하는 부분")으로 표시 — 본 시나리오에 섞이면 사용자가 어느 쪽을 믿어야 할지 헷갈림.

**마찰점 (F5)**: 시나리오에 개발자 용어(예: "POST `/api/v1/...`")가 잔존하면 P2의 신뢰가 즉시 무너짐. 이 화면은 **비개발자 검수**가 필수 기준선.

**페르소나 메모**:
- P1: 코드 근거 링크가 진짜 그 시나리오의 출처인지 검수하는 데 시간 씀.
- P2: 화면을 위에서 아래로 읽기만 하면 이해되어야 함. 막히는 곳이 있으면 그 자체가 표현 결함의 신호.

### S09 — 종단 의존성

[`s09-feature-dependencies.svg`](../wireframes/s09-feature-dependencies.svg)

| 항목 | 내용 |
|------|------|
| **사용자 의도** | "이 feature가 어떤 인프라/데이터/모듈/인터페이스에 의존하는지 한눈에 보고 싶다." |
| **주요 행동** | (1) 의존성 그래프 훑어보기 (2) 카테고리별 의존성 목록(인프라·데이터 모델·아키텍처 레이어·프레임워크·미들웨어·로직 모듈·외부 인터페이스) 펼쳐 보기 (3) 각 의존성의 코드 근거 확인 |
| **시스템 응답** | 그래프 + 카테고리 두 가지 형태로 노출 · 각 의존성에 코드 근거 첨부 · 의존성은 *별도 구조화 데이터*로 저장돼 역방향 질의 가능 |
| **이 화면의 끝** | 사용자는 영향 범위를 인지한 상태로 [S08](../wireframes/s08-feature-acceptance.svg)/[S10](../wireframes/s10-llm-edit.svg)로 돌아감 |

**이 화면의 약속** (V5, V2 / AC2.4, AC2.5):
- 의존성은 **단순 텍스트가 아니라 데이터**로 저장됨 → 다른 feature와 비교/교차 조회 가능.
- 의존성에 카테고리(7종 — 인프라·데이터·아키텍처·프레임워크·미들웨어·로직·인터페이스)가 명시됨.
- 역방향 질의 가능: "데이터 모델 X를 사용하는 feature 전부", "미들웨어 Y에 영향받는 feature 전부".

**의존성을 보는 두 가지 길**:
- *순방향 (이 화면)*: "이 feature는 무엇에 의존하는가?" — 카테고리별 노출.
- *역방향 ([플로우 4](./04-continuous-maintenance.md)에서 등장)*: "이 데이터 모델/미들웨어에 영향받는 feature는?" — AC2.5의 별도 데이터가 만들어 둔 질의 능력.

**페르소나 메모**:
- P1: 평소 *알고 있던 의존성을 처음 한 자리에서 본다* 는 경험. V5를 처음 체감하는 순간.
- P2: 그래프보다 카테고리별 목록이 더 친화적. "외부 인터페이스 = 어디서 호출됨" 같은 자연어 라벨이 중요.

### S10 — LLM 보조 편집

[`s10-llm-edit.svg`](../wireframes/s10-llm-edit.svg)

| 항목 | 내용 |
|------|------|
| **사용자 의도** | "이 시나리오 한 줄을 더 명확하게 / 에러 케이스를 보강하고 싶은데 직접 쓰는 건 번거롭다. LLM에게 자연어로 시키고 결과만 보고 결정하고 싶다." |
| **주요 행동** | (1) 자연어 한 줄 입력("이 시나리오 더 명확하게", "에러 케이스 추가" 등) (2) 제안된 diff 검토 (3) 승인/거부 |
| **시스템 응답** | LLM이 변경 제안을 diff로 먼저 보여줌 · 승인 전에는 적용되지 않음 · 거부된 제안은 기록되어 다음 제안에서 회피 · 변경 이력에 "사용자 by LLM" 출처로 기록 |
| **이 화면의 끝** | diff가 적용되거나(승인) 폐기됨(거부). 사용자는 [S08](../wireframes/s08-feature-acceptance.svg)으로 돌아감 |

**이 화면의 약속** (V3, V4, V7 / AC3.1):
- LLM 제안은 **항상 diff로 먼저 보여준다**. 승인 없이 적용되는 경로 없음.
- 거부된 제안은 기록되어 동일 feature에 같은 패턴의 제안이 반복되지 않음.
- 모든 변경은 출처(자동 / 사용자 by LLM / 사용자 직접)와 함께 이력으로 보존 (AC3.4 — [플로우 4](./04-continuous-maintenance.md)에서 활용).

**3탭 룰** (마찰점 F8): 모바일에서 "수정 요청 → diff 확인 → 승인/거부"가 **3탭 이내**에 끝나야 합니다. 키보드 노출, 모달 중첩, 스크롤 깊이를 모두 검수해야 함. 이게 깨지는 순간 사용자는 "직접 쓰는 게 빠르겠는데"가 되어 LLM 보조 편집을 안 쓰게 됩니다.

**페르소나 메모**:
- P1: 표현 디테일을 빠르게 정돈하는 용도.
- P2: 자기가 발견한 모호함을 자연어로 LLM에게 넘기는 용도. "어떻게 쓰면 좋을지 모르겠지만 이상하다"는 감각을 그대로 LLM에게 위임 가능.

## 이 플로우의 시나리오

**14:35 · 카페** — Pre는 12개 feature 중 "사용자 비밀번호 재설정"을 골라 [S08](../wireframes/s08-feature-acceptance.svg)에 들어감. 시나리오 4개가 Given-When-Then으로 적혀 있고, 각 시나리오 끝에 코드 위치 링크가 작은 글씨로 달림. 비개발자도 읽힐 수 있게 *"잘못된 토큰으로 시도했을 때 / 만료된 토큰으로 시도했을 때"* 같은 자연어.

**14:38** — [S09](../wireframes/s09-feature-dependencies.svg). 이 feature가 SMTP 서비스 · `users` 테이블 · 인증 미들웨어에 의존함을 그래프로 봄. Pre는 평소 모르고 있던 게 아니라 *"한 자리에서 본 적이 없었다"*. V5를 처음 체감.

**14:41** — [S10](../wireframes/s10-llm-edit.svg). 시나리오 2번이 약간 모호해 보여 *"에러 케이스 더 명확하게 써줘"* 입력. diff에 추가 시나리오 *"만료된 재설정 토큰으로 시도했을 때"*. 승인 (3탭 이내 종료). 변경 이력에 "사용자 by LLM" 한 줄 기록.

**14:43** — Pre는 12개 중 1개만 처리한 상태로 앱을 닫음. 나머지 11개는 평일에. [플로우 4](./04-continuous-maintenance.md)는 코드 push가 일어나는 시점부터 자동 시작.

이 플로우 안에서 사용자가 들인 능동 시간 (1개 feature): **약 8분**.

## P2가 이 플로우에서 가장 자주 진입하는 경로

```
홈 → 저장소 → feature 목록 → 한 feature 선택
                                    ↓
                                  [S08] 인수 시나리오 읽기
                                    ↓
                                  (필요 시) [S09] 의존성 보기
                                    ↓
                                  (이상 발견 시) [S10] 자연어 편집 요청
                                    ↓
                                  P1이 변경 이력에서 검토
```

P2는 *분석을 트리거하지 않고*, *후보를 승인하지 않으며*, **확정된 feature 위에서만** 검토와 편집 제안을 합니다. 이 분리가 권한 격리(AC4.7)와 맞물려 운영적으로도 안전.

## 플로우 종료 조건과 다음 플로우

이 플로우는 **사용자가 한 feature의 검토를 끝내는 시점**에 종료됩니다(혹은 여러 feature를 차례로 처리하는 반복 진입). 종료 후:

- 코드 push가 발생하면 **[플로우 4 — 지속적 유지](./04-continuous-maintenance.md)** 가 자동 트리거.
- 이 플로우에서의 모든 편집은 **이력**으로 보존되어, 플로우 4의 충돌 처리/이력 회고에서 활용됨 (AC3.4).
