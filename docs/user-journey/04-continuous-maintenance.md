# 플로우 4 — 지속적 유지 (Continuous Maintenance)

> 코드가 변경된 뒤 분석이 다시 돌고, 자동 결과가 사용자 편집과 부딪히는 순간을 다루는 흐름. 처음 분석 이후의 *시간이 흐른 뒤* 발생하는 모든 흐름의 묶음.
>
> 페르소나·전체 개요는 [`README.md`](./README.md)를 보세요. 이 문서는 **이 플로우 안의 시나리오 단위 흐름**을 다룹니다.

## 의도와 맥락

**사용자 의도**: "코드가 바뀌었다. 이 도구가 알아서 따라오고 있고, 내 편집과 충돌이 나면 나에게 물어본다."

이 플로우는 다른 플로우들과 다르게 **고유 화면이 없습니다**. 화면은 [플로우 2](./02-discovery.md)와 [플로우 3](./03-feature-documents.md)의 화면을 *시점만 다르게* 재진입합니다. 따라서 화면 단위가 아니라 **재진입 시나리오 단위**로 정리합니다.

이 플로우의 핵심 약속 한 줄:

> **자동 분석은 사용자의 명시적 결정 없이 사용자 편집을 절대 덮어쓰지 않는다.**

이 약속이 깨지는 순간 사용자가 도구를 신뢰하는 근거가 사라져요 (V4 + AC3.5).

**주된 페르소나**: P1, P2 둘 다. 단, 충돌 해결의 *최종 결정권*은 보통 P1.

## 4개 재진입 시나리오

```
[시나리오 A]              [시나리오 B]                 [시나리오 C]                [시나리오 D]
재분석 트리거       →     자동 vs 편집 충돌      →    삭제된 feature 재발견   →    변경 이력 회고
(AC2.6)                   (AC3.5)                      (AC3.3)                      (AC3.4)
```

각 시나리오는 독립적으로 발생할 수 있고, 서로 결합돼 함께 발생하기도 합니다 (예: 재분석 → 충돌 → 충돌 화면에서 LLM 보조 병합).

---

## 시나리오 A — 재분석 트리거와 diff 노출

| 항목 | 내용 |
|------|------|
| **언제 발생** | 코드 변경(push, 새 커밋, 명시적 재분석 요청) 후. |
| **사용자 의도** | "이번 변경이 어느 feature의 표현/의존성을 바꿨는지 알고 싶다." |
| **재진입 화면** | [S04](../wireframes/s04-analysis-progress.svg)(분석 진행) → [S07](../wireframes/s07-feature-candidates.svg)(후보 변화) → [S08](../wireframes/s08-feature-acceptance.svg)/[S09](../wireframes/s09-feature-dependencies.svg) (개별 feature의 diff) |
| **시스템 응답** | 영향받는 feature의 인수 시나리오·의존성이 갱신됨 · 이전 버전과의 diff가 표시됨 · 변경되지 않은 feature는 그대로 유지됨 |
| **이 시나리오의 끝** | 사용자가 변경 사항을 인지하고, 필요한 feature에 대해 시나리오 B/D로 분기 가능 |

**약속 (V4, V5 / AC2.6)**: 동일 feature를 두 시점에 분석하면 *변경된 시나리오 / 추가·제거된 의존성*이 diff로 명확히 노출됨.

**페르소나 메모**:
- P1: diff를 빠르게 훑어 영향 범위를 파악. "내가 한 코드 변경이 의도대로 feature에 반영됐는가?"의 즉답이 됨.
- P2: 보통 알림으로 diff 요약을 받고 필요 항목만 확인.

## 시나리오 B — 자동 결과와 사용자 편집의 충돌

| 항목 | 내용 |
|------|------|
| **언제 발생** | 자동 재분석 결과가 [플로우 3](./03-feature-documents.md)의 [S10](../wireframes/s10-llm-edit.svg)에서 한 사용자 편집과 부딪힐 때. |
| **사용자 의도** | "어느 쪽을 살릴지 내가 정하고 싶다. 자동으로 덮어쓰지 마라." |
| **재진입 화면** | 충돌 표시 + 사용자 편집 vs 자동 결과 비교 화면 (별도 화면이 아니라 [S08](../wireframes/s08-feature-acceptance.svg)에 충돌 배너로 표시됨) → 필요 시 [S10](../wireframes/s10-llm-edit.svg)로 LLM 보조 병합 |
| **시스템 응답** | 충돌이 표시되며, 사용자는 (a) 자동 결과 채택 (b) 편집 유지 (c) LLM 보조 병합 중 선택 · 사용자 결정 없이는 자동 결과로 덮이지 않음 |
| **이 시나리오의 끝** | 한 feature의 충돌이 해소되며 변경 이력에 한 줄 추가 |

**약속 (V4, V7 / AC3.5)**:
- 충돌 발생 시 **자동으로 한쪽을 채택하지 않는다**.
- 결정 옵션은 항상 *3개*: 자동 결과 / 편집 유지 / LLM 보조 병합.
- 충돌이 해소되지 않은 feature는 명확히 표시되어, 다음 자동 분석에서도 그대로 충돌 상태 유지.

**마찰점 (F6)**: 자동 분석이 사용자 편집을 덮어쓰는 일이 한 번이라도 발생하면 사용자는 LLM 보조 편집(S10)을 더 이상 쓰지 않게 됨. 충돌 검출 로직은 *과한 쪽이 안전*.

**페르소나 메모**:
- P1: "내가 손댔던 시나리오를 자동이 다르게 고치려 한다"는 사실 자체가 신호. 보통 (b) 편집 유지를 우선 선택, 자동 결과의 일부만 (c) 병합으로 흡수.
- P2: 충돌 결정에 보통 직접 관여하지 않음. P1에게 권한이 있는 모드.

## 시나리오 C — 거부했던 feature가 코드에 다시 잡힘

| 항목 | 내용 |
|------|------|
| **언제 발생** | [플로우 2](./02-discovery.md)의 [S07](../wireframes/s07-feature-candidates.svg)에서 거부했던 후보가, 다음 자동 분석에서 같은 위치에서 다시 후보로 잡힐 때. |
| **사용자 의도** | "전에 거부한 거다. 다시 거부할지 / 이번엔 채택할지 결정하고 싶다." |
| **재진입 화면** | [S07](../wireframes/s07-feature-candidates.svg) — 해당 후보에 *"이전에 거부된 항목입니다 + 거부 사유"* 가 표시됨 |
| **시스템 응답** | 재분석에서 같은 후보가 다시 잡혀도 *거부 사유와 함께* 사용자에게 노출 · 자동 채택되지 않음 |
| **이 시나리오의 끝** | 다시 거부 / 이번엔 채택 / 사유 갱신 중 사용자가 선택 |

**약속 (V4 / AC1.4 + AC3.3)**:
- 거부 사유는 다음 자동 분석에서 보존되고 다시 노출됨.
- 삭제된 feature 자체는 즉시 영구 삭제되지 않고 *soft delete* 상태로 일정 기간 복구 가능.
- 이 두 약속이 합쳐져 **사용자 의도와 자동 결과의 충돌 방지** 가 됨.

**마찰점 (F7)**: 거부 사유가 보존되지 않으면 *같은 결정*을 사용자가 매 분석마다 반복해야 함. 사유의 *명료함*은 다음 분석의 품질을 좌우.

**페르소나 메모**:
- P1: 보통 같은 결정을 반복(거부 유지). 다만 코드가 의미 있게 바뀌었다면 채택으로 전환되는 경우도.

## 시나리오 D — 변경 이력 회고

| 항목 | 내용 |
|------|------|
| **언제 발생** | 사용자가 한 feature가 어떻게 진화했는지 보고 싶을 때, 또는 잘못된 변경을 되돌리고 싶을 때. |
| **사용자 의도** | "이 feature의 임의 시점 상태를 복원하거나 변경 출처를 확인하고 싶다." |
| **재진입 화면** | [S08](../wireframes/s08-feature-acceptance.svg) 안의 변경 이력 진입점 → 이력 화면 (별도 wireframe 미정, 추가 필요) |
| **시스템 응답** | 모든 변경의 출처가 *자동 분석 / 사용자 by LLM / 사용자 직접* 중 하나로 표시 · 임의 시점의 상태 복원 가능 |
| **이 시나리오의 끝** | 사용자가 이력을 인지 또는 특정 시점으로 복원 |

**약속 (V4 / AC3.4)**:
- 모든 변경은 *출처와 함께* 이력으로 보존됨. 출처는 3종.
- 임의 시점의 상태로 복원 가능.
- 출처 정보는 모바일 화면에서 *한눈에* 확인 가능.

**누락 항목**: 이 시나리오는 현재 와이어프레임이 없습니다. 향후 [`wireframes/`](../wireframes/)에 변경 이력 화면(S11 후보)을 추가할 때 [README §6](./README.md#6-갱신-정책)의 갱신 정책에 따라 본 문서도 갱신해야 함.

**페르소나 메모**:
- P1: 자기 또는 자동의 변경을 추적하는 데 사용. 잘못된 자동 분석 결과를 빠르게 되돌리는 도구.
- P2: 거의 진입하지 않음. P1이 회고용으로 사용.

## 플로우 종료 조건과 사이클

이 플로우는 **종료되지 않습니다**. 코드가 살아 있는 한 시나리오 A~D가 비주기적으로 반복돼요. 다만 한 번의 재분석 사이클은:

- 시나리오 A로 시작 → (필요 시) B / C / D로 분기 → 변경 이력에 한 줄 추가 → 다음 코드 변경 대기.

이 사이클이 *예측 가능하고 사용자가 통제하는* 형태로 유지되는 한, V4(살아있는 문서)는 약속을 지키고 있는 것.
