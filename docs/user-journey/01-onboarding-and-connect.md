# 플로우 1 — 진입과 연결 (Onboarding & Connect)

> 사용자가 FeatureDoc에 처음 들어와 자격증명을 등록하고, 자기 저장소를 안전하게 연결한 뒤 분석을 트리거하기까지.
>
> 페르소나·전체 개요는 [`README.md`](./README.md)를 보세요. 이 문서는 **이 플로우 안의 화면 단위 흐름**만 다룹니다.

## 의도와 맥락

**사용자 의도**: "내가 가진 저장소를 이 도구가 안전하게 들여다볼 수 있게 만들고 싶다."

이 플로우는 사용자가 FeatureDoc을 **가장 의심하는 시간**입니다. GitHub App을 설치하고 저장소 접근을 허용하는 행위 자체가 위험 감각을 자극해요. 그래서 모든 화면의 마이크로카피는 "읽기 전용 최소 권한만 요청합니다 / 당신이 고른 저장소에만 접근합니다 / 언제든 설치를 해제할 수 있습니다 / LLM 호출은 당신의 키로 일어납니다"를 짧고 명확하게 반복해야 합니다.

**주된 페르소나**: P1 (Pre)만 들어옵니다. P2는 이 플로우에 거의 진입하지 않아요.

## 화면 흐름

```
[S01]                  [S02]                       [S03]
자격증명 등록    →     저장소 목록 (홈)      →     저장소 연결 + 분석 시작
(첫 진입)              (재방문 시 시작점)          (URL/브랜치/비용 안내)
```

화면 흐름은 분기 가능: 첫 사용자는 S01 → S03 직행, 재방문 사용자는 S02에서 시작하여 새 저장소를 추가하거나 기존 분석으로 이동합니다.

## 화면별 상세

### S01 — 자격증명 등록

[`s01-credentials-setup.svg`](../wireframes/s01-credentials-setup.svg)

| 항목 | 내용 |
|------|------|
| **사용자 의도** | "내 저장소를 안전하게 연결하고 LLM API Key를 등록하고 싶다." |
| **주요 행동** | (1) GitHub App 설치 시작 (2) 요청 권한 범위 확인 후 GitHub에서 접근할 저장소 선택 (3) LLM API Key 입력 (4) 등록 |
| **시스템 응답** | App 설치 후 접근 허용된 저장소를 표시 · LLM Key는 평문 미저장·식별자만 표시 · 언제든 설치 해제/키 교체 가능 안내 |
| **이 화면의 끝** | 저장소 접근과 LLM Key가 준비되고, 사용자는 S02 또는 S03으로 이동 |

**이 화면의 약속** (V6 / AC4.1, AC4.2, AC4.3):
- 자격증명은 평문으로 저장되지 않는다.
- 메모리상에서도 호출 직전에만 복호화된다.
- 폐기 시 진행 중 분석은 명시된 정책(중단 / 현재 호출까지 완료)에 따라 처리된다.

**마찰점 (F1)**: 외부 App에 저장소 접근을 허용하는 행위 자체가 심리적 장벽. 요청 권한이 읽기 전용 최소 범위라는 점과 사용자가 접근할 저장소를 직접 고른다는 점이 화면에 *눈에 보이는* 형태로 있어야 합니다 — 약관 깊은 곳이 아니라 설치 버튼 바로 위에.

**페르소나 메모**:
- P1: 가장 경계하는 순간. 한 번 신뢰가 깨지면 회복 어려움.
- P2: 이 화면에 거의 들어오지 않음 (이미 등록된 인스턴스를 공유받음).

### S02 — 저장소 목록 (홈)

[`s02-home-repositories.svg`](../wireframes/s02-home-repositories.svg)

| 항목 | 내용 |
|------|------|
| **사용자 의도** | "내가 연결한 저장소들의 분석 상태를 한눈에 보고, 들어가거나 새로 추가하고 싶다." |
| **주요 행동** | (1) 연결된 저장소 목록 훑기 (2) 각 저장소의 분석 상태(미분석/진행 중/완료/실패) 확인 (3) 새 저장소 추가 또는 기존 저장소 진입 |
| **시스템 응답** | 진행 중 작업이 있으면 진행률 노출 · 실패 작업이 있으면 재시도 진입 가능 · 자격증명 미등록 상태에서는 추가 시 S01로 우회 |
| **이 화면의 끝** | 새 저장소 추가 흐름(S03)으로 진입하거나, 기존 저장소의 결과 흐름(플로우 2/3)으로 진입 |

**이 화면의 역할** (V7 / AC1.5, AC4.6):
- 재방문 사용자의 **시작점**. 짧은 세션에서 즉시 어디로 갈지 결정할 수 있어야 함.
- 모든 진행 중 작업의 *현재 상태*를 화면에서 봐야 한다 (백그라운드 분석을 신뢰하기 위한 근거).

**마찰점 없음**: 이 화면 자체보다는 진입 후 분기에서 마찰이 생기는 *허브* 성격.

**페르소나 메모**:
- P1: 가장 자주 진입하는 화면. 빠른 스캔 가능성이 핵심.
- P2: 자기에게 권한이 있는 저장소만 보임. 공유받지 않은 저장소는 노출되지 않음 (AC4.7 배경 약속).

### S03 — 저장소 연결 + 분석 시작

[`s03-connect-repository.svg`](../wireframes/s03-connect-repository.svg)

| 항목 | 내용 |
|------|------|
| **사용자 의도** | "이 저장소·이 브랜치를 분석에 걸고 싶다. 단, 비용을 미리 알고 싶다." |
| **주요 행동** | (1) 저장소 URL 입력 (2) 분석 대상 브랜치 지정 (3) 예상 LLM 호출/비용 안내 확인 (4) "분석 시작" 탭 |
| **시스템 응답** | URL 검증 · App 설치 범위에 포함된 저장소인지 확인 · pre-flight 추정 (예상 호출 횟수/비용) · 큐에 비동기 작업 등록 |
| **이 화면의 끝** | "분석을 시작했습니다" 상태로 전환되며, 사용자는 자유롭게 앱을 닫거나 S04(분석 진행)로 이동 |

**이 화면의 약속** (V6, V7, V8 / AC1.1, AC4.6):
- 분석은 **명시적 트리거**로만 시작한다 (자동으로 시작되지 않는다).
- 사용자는 시작 직전에 *예상 호출 규모와 추정 비용 범위*를 본다.
- App 접근 범위에 없는 저장소면 단순 "실패"가 아니라 **App 설치 범위에 추가하는 경로**를 안내한다.

**마찰점 (F2)**: 비용 안내가 모호하면 분석 시작을 미루게 됨. 예상 호출 횟수와 추정 비용 범위를 시작 버튼 *바로 위에* 명시. 안내 없이 시작 가능한 경로가 있으면 안 됨.

**오류 시 흐름**:
- 접근 범위 밖 → "이 저장소에 접근할 수 없습니다 · GitHub App 설치 범위에 이 저장소를 추가해야 합니다 · App 설치 관리 열기" 로 S01로 우회 가능.
- URL 오타 → 즉시 검증 실패 표시.
- 큐 등록 실패 (운영 측 문제) → 재시도 가능, AC1.5의 단계별 재시도 메커니즘이 여기서도 작동.

**페르소나 메모**:
- P1: 비용 안내 + 권한 안내가 충분히 명확하면 한 번에 통과.
- P2: 보통 이 화면에 들어오지 않음.

## 이 플로우의 시나리오

**14:02 · 카페** — Pre는 폰을 꺼내 [S01](../wireframes/s01-credentials-setup.svg)을 연다. 설치 버튼 바로 위에 *"읽기 전용 최소 권한만 요청합니다"* 한 줄. 약간 안심하고 GitHub App 설치 → GitHub 설치 페이지에서 자기 사이드 프로젝트 저장소 하나만 골라 접근 허용. 앱으로 돌아와 같은 화면에서 LLM API Key까지 등록.

**14:04** — [S02](../wireframes/s02-home-repositories.svg)로 자동 이동, 비어 있는 목록. "+" 탭.

**14:05** — [S03](../wireframes/s03-connect-repository.svg)에 자기 사이드 프로젝트 URL과 `main` 입력. 시작 버튼 위에 *"예상 LLM 호출 ~120회 / 추정 비용 ~$0.4"*. "이 정도면 괜찮네" 하고 시작.

**14:06** — "분석을 시작했습니다 · 알림으로 알려드릴게요" 상태로 전환. Pre는 앱을 닫고 일행과 이야기하러 감. 이 시점부터 [플로우 2](./02-discovery.md)가 백그라운드에서 시작됨.

이 플로우 안에서 사용자가 들인 능동 시간: **약 4분** (S01에서 처음이라 가장 오래 걸림).
