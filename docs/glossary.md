## 사용자 유형

### T2: 최종 사용자
- **유형**: 제품고유 (혼동 위험 큼 — T3 "사용자"와 반드시 구분)
- **정의**: 분석 대상 GitHub 저장소가 제공하는 제품을 실제로 쓰는 사람. FeatureDoc 자체의 사용자가 아니라, FeatureDoc이 분석하는 외부 제품의 사용 주체이다.
- **반의/대비 용어**: T3 사용자
- **사용 문맥**: feature가 누구의 관점에서 표현되어야 하는가를 말할 때(인수 테스트의 시점). 가치 문서의 정체성 정의, V3.
- **정의 출처/소유자**: docs/values.md (제품 소유자 TBD)

### T3: 사용자
- **유형**: 사용자유형 (혼동 위험 큼 — T2 "최종 사용자"와 반드시 구분)
- **정의**: FeatureDoc 자체를 사용하는 사람. 자신의 GitHub 저장소를 분석하려는 개발자·PM 등. 자격증명 등록, 분석 트리거, feature 문서 편집의 주체.
- **반의/대비 용어**: T2 최종 사용자
- **사용 문맥**: 거의 모든 PRD에서 등장. 자격증명 등록·교체·폐기, 분석 트리거, LLM 보조 수정 등.
- **정의 출처/소유자**: 가치 문서 + PRD 전반 (제품 소유자 TBD)

### T4: 제품 소유자

- **정의 출처/소유자**: docs/values.md V3

## 분석·표현의 핵심 개념

### T1: Feature
- **유형**: 제품고유
- **정의**: FeatureDoc이 다루는 1차 단위. 분석 대상 저장소가 최종 사용자에게 노출하는 하나의 기능. 일상어 "기능"과 달리, 반드시 최종 사용자 관점에서 식별 가능하고 인수 테스트로 표현 가능한 단위만을 가리킨다.
- **반의/대비 용어**: T8 횡단 관심사 (feature는 종단 단위)
- **사용 문맥**: 분석 파이프라인의 산출물 단위, 인수 테스트·의존성 데이터의 키. 가치 문서, 모든 PRD에 등장.
- **정의 출처/소유자**: docs/values.md 정체성 + PRD-1 AC1.4

### T8: 횡단 관심사
- **유형**: 도메인
- **풀이**: Cross-cutting concerns
- **정의**: 코드베이스를 통째로 가로지르는 구조적 요소. 인프라(배포 매니페스트/IaC), 저장소 구조(monorepo 여부 등), 아키텍처 패턴, 프레임워크/런타임, 미들웨어 등을 포함한다. 특정 feature에 묶이지 않고 전체에 영향을 주는 것들.
- **반의/대비 용어**: T9 종단 의존성
- **사용 문맥**: V2, PRD-1 AC1.2.
- **정의 출처/소유자**: PRD-1 AC1.2

### T9: 종단 의존성
- **유형**: 제품고유
- **정의**: 한 feature가 의존하는 인프라·데이터 모델·아키텍처 레이어·프레임워크/라이브러리·미들웨어·핵심 로직 모듈·외부 인터페이스(API/UI/CLI 진입점)를 끝에서 끝까지 식별한 것. 별도의 구조화된 데이터(feature ID 키)로 보존된다.
- **반의/대비 용어**: T8 횡단 관심사
- **사용 문맥**: V5, V2, PRD-2 AC2.4·AC2.5.
- **정의 출처/소유자**: PRD-2 AC2.4

### T10: 횡단 분석
- **유형**: 제품고유
- **정의**: 코드베이스에서 횡단 관심사(T8)를 추출하는 분석 단계. 분석 파이프라인의 1단계.
- **사용 문맥**: PRD-1 AC1.2, 테스트 시나리오 1·3.
- **정의 출처/소유자**: PRD-1 AC1.2

### T11: 종단 분석
- **유형**: 제품고유
- **정의**: 한 feature에서 시작해 그것이 의존하는 모든 계층을 끝까지 따라가는 분석. 종단 의존성(T9) 추출의 기반.
- **사용 문맥**: V2, PRD-2 AC2.4.
- **정의 출처/소유자**: PRD-2 AC2.4 + 가치 문서 V2

### T12: 인수 기준
- **유형**: 도메인 / 약어
- **풀이**: Acceptance Criteria (약칭 AC)
- **정의**: feature가 만족해야 할 조건 한 개. "주어진 ~ / ~ 했을 때 / ~ 해야 한다" 형식으로 표현된다. PRD 안의 단위이자, 인수 테스트 시나리오의 단위.
- **사용 문맥**: 모든 PRD, doc-tracker.md.
- **정의 출처/소유자**: PRD-2 AC2.1, README

### T13: 인수 테스트
- **유형**: 제품고유
- **정의**: 한 feature가 최종 사용자 관점에서 무엇을 해야 하는지 시나리오 단위로 표현한 문서. 1개 feature당 1개의 인수 테스트 문서가 생성된다. 개발자 용어가 아닌 최종 사용자가 알아볼 수 있는 언어로 작성된다.
- **사용 문맥**: V3, PRD-2 AC2.3, PRD-3 거의 전체.
- **정의 출처/소유자**: PRD-2 AC2.3

### T14: 탐색 전략
- **유형**: 제품고유
- **정의**: "이 코드베이스에서 feature가 어디서 어떻게 발견될 수 있는가"에 대한, 사람이 읽고 편집할 수 있는 규약. 진입점 파일 패턴, 라우팅 정의 위치, UI 컴포넌트 경로, API 엔드포인트, CLI 명령 등이 명시된다. 사용자가 검토·수정·승인한 전략만 다음 단계의 입력이 된다.
- **사용 문맥**: PRD-1 AC1.3·AC1.4.
- **정의 출처/소유자**: PRD-1 AC1.3

### T15: feature 후보
- **유형**: 제품고유
- **정의**: 탐색 전략을 통해 추출되었으나 아직 사용자가 승인하지 않은 feature. 사용자는 후보를 승인/거부/병합/이름 변경할 수 있다. 거부된 후보의 사유는 다음 분석 시 참고되도록 기록된다.
- **사용 문맥**: PRD-1 AC1.4, 시나리오 7.
- **정의 출처/소유자**: PRD-1 AC1.4

### T16: 분석 파이프라인
- **유형**: 제품고유
- **정의**: "횡단 관심사 분석 → feature 탐색 전략 생성 → feature 후보 추출"의 3단계로 구성된, 코드베이스 1회 분석의 입력·출력·실행 모델. PRD-1의 대상 자체.
- **사용 문맥**: README, PRD-1 전체, doc-tracker.md.
- **정의 출처/소유자**: PRD-1

## 문서 체계 관련

### T17: 살아있는 문서
- **유형**: 제품고유
- **정의**: 코드 변경에 뒤처지지 않고, 실제 로직과 테스트 코드를 근거로 LLM이 갱신해 코드와 함께 진화하는 feature 문서를 가리키는 표현. V4의 핵심 표현.
- **사용 문맥**: V4, AC2.6.
- **정의 출처/소유자**: docs/values.md V4

### T18: 변경 출처
- **유형**: 제품고유
- **정의**: feature 문서의 변경이 어떤 경로로 발생했는지 구분하는 표시. 다음 세 값을 가진다 — `자동`(코드 재분석에 의한 변경), `사용자 by LLM`(사용자가 LLM 보조로 한 변경), `사용자 직접`(LLM을 거치지 않고 직접 편집). 변경 이력에 함께 보존된다.
- **사용 문맥**: PRD-3 AC3.4, 시나리오 1·8.
- **정의 출처/소유자**: PRD-3 AC3.4

### T19: 보관소
- **유형**: 제품고유
- **정의**: 사용자가 삭제한 feature를 즉시 영구 제거하지 않고 일정 기간 보존하는 영역. soft delete의 저장 위치. 자동 재발견 시 "이전에 거부/삭제된 항목"으로 참조되어 사용자 의도와 충돌하지 않게 한다.
- **사용 문맥**: PRD-3 AC3.3.
- **정의 출처/소유자**: PRD-3 AC3.3

### T20: 코드 근거
- **유형**: 제품고유
- **정의**: 분석·제안의 출처가 된 코드 위치(파일 경로, 심볼 등). 모든 LLM 산출물(횡단 관심사 항목, 인수 기준, feature 후보, 의존성 등)에 첨부된다. 코드 근거가 없으면 시스템은 결과를 강제로 만들어내지 않는다 (AC3.2의 "근거 없음" 표시).
- **사용 문맥**: AC1.2, AC1.4, AC2.1, AC2.2, AC2.4, AC3.2.
- **정의 출처/소유자**: PRD-1 AC1.2 + PRD-3 AC3.2

### T21: 가치 문서
- **유형**: 제품고유
- **정의**: 제품이 제공하려는 가치(V1~V8)를 정의한 최상위 문서. 모든 PRD/AC/테스트는 자신이 어떤 가치를 달성하는지 명시한다.
- **사용 문맥**: docs/values.md, doc-tracker.md, README.
- **정의 출처/소유자**: docs/values.md

### T36: 제품 가치
- **유형**: 제품고유
- **정의**: 제품이 사용자에게 약속하는 추상적·구체적 효용. 본 제품에서는 V1~V8로 번호화되어 가치 문서에 정의되며, 모든 PRD/AC/테스트는 자신이 달성하는 가치를 명시한다. 가치와 연결되지 않은 문서는 위험으로 진단된다.
- **사용 문맥**: docs/values.md, 모든 PRD의 "달성 가치" 섹션, doc-tracker.md.
- **정의 출처/소유자**: docs/values.md

### T30: PRD
- **유형**: 약어
- **풀이**: Product Requirements Document
- **정의**: 가치를 만족시키기 위한 기능 요구사항을 acceptance criteria 단위로 기술한 문서. FeatureDoc 문서 체계에서 PRD는 가치(V)와 테스트 사이의 중간 계층이다.
- **사용 문맥**: docs/prd/, doc-tracker.md, README.
- **정의 출처/소유자**: README, doc-tracker.md

## 코드베이스 구성요소 (의존성 카테고리)

### T22: 미들웨어
- **유형**: 도메인
- **정의**: 인증·로깅·캐시 등 요청 처리 흐름에 가로로 끼어드는 횡단 관심사 컴포넌트. 횡단 관심사(T8)의 한 항목이자, 종단 의존성(T9)의 한 카테고리.
- **사용 문맥**: AC1.2(추출 대상), AC2.4(의존성 카테고리), 시나리오 5·6.
- **정의 출처/소유자**: PRD-1 AC1.2 + PRD-2 AC2.4

### T23: 인프라
- **유형**: 도메인
- **정의**: 코드베이스 분석 맥락에서, 배포 매니페스트/IaC/외부 서비스/큐/DB 등 시스템 동작을 위한 외부적 요소. 횡단 관심사·종단 의존성 양쪽에서 카테고리로 등장한다.
- **사용 문맥**: AC1.2, AC2.4.
- **정의 출처/소유자**: PRD-1 AC1.2 + PRD-2 AC2.4

### T24: 데이터 모델
- **유형**: 도메인
- **정의**: feature가 의존하는 엔티티/스키마 정의. 종단 의존성의 한 카테고리이며, 역방향 질의("데이터 모델 X를 사용하는 feature 전부")의 키가 된다.
- **사용 문맥**: PRD-2 AC2.4·AC2.5, 시나리오 6.
- **정의 출처/소유자**: PRD-2 AC2.4

### T34: IaC
- **유형**: 약어
- **풀이**: Infrastructure as Code
- **정의**: 인프라 정의를 코드로 표현한 것(예: Terraform, Helm 차트, k8s 매니페스트). 횡단 관심사의 인프라 항목으로 추출 대상이 된다.
- **사용 문맥**: PRD-1 AC1.2.
- **정의 출처/소유자**: PRD-1 AC1.2

### T35: monorepo
- **유형**: 도메인
- **정의**: 여러 모듈/서비스를 하나의 저장소에 함께 두는 저장소 구조. 횡단 관심사 분석에서 저장소 구조 항목으로 식별된다.
- **사용 문맥**: PRD-1 AC1.2.
- **정의 출처/소유자**: PRD-1 AC1.2

## 자격증명·플랫폼

### T25: GitHub Token
- **유형**: 외부표준
- **풀이**: GitHub Personal Access Token (PAT) 또는 fine-grained token
- **정의**: 사용자(T3)가 본인의 GitHub 저장소에 FeatureDoc이 접근하도록 하기 위해 등록하는 인증 토큰. 평문으로 저장되지 않으며, 사용자가 등록·교체·폐기를 통제한다.

## 용어 ID 정책

- 용어 ID(T1, T2, ...)는 **변경되지 않는 고정 식별자**입니다. 용어 이름이 바뀌어도 ID는 유지합니다.
- 용어가 폐기되더라도 ID는 "사용 안 함"으로 표시하고, 같은 ID를 새 용어에 재부여하지 않습니다.
- 다른 문서에서 용어를 참조할 때는 본 문서의 정식 용어 이름을 사용합니다(필요 시 ID 병기).
