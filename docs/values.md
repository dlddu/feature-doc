## 제품 정체성

GitHub 저장소에 연결되어, 그 저장소가 제공하는 **최종 사용자용 기능(feature)** 을 LLM의 도움으로 발견·표현·관리하기 쉽게 만드는 도구. 코드베이스를 횡단(인프라/아키텍처/프레임워크)과 종단(feature → 의존성)으로 모두 분석하여, 코드와 항상 함께 살아 움직이는 feature 문서 체계를 유지하는 것이 본 제품의 본질이다.

## 제품 가치

### V1: 코드에서 시작하는 feature 발견 가능성
- **유형**: 추상적
- **설명**: 사용자가 직접 정리하지 않아도, 코드베이스로부터 최종 사용자용 기능을 자동으로 발견할 수 있어야 한다. 발견의 근거가 코드 자체이므로 누락과 환상이 줄어든다.
- **요구사항 근거**: #2, #6, #7

### V2: 횡단·종단 양방향 이해
- **유형**: 추상적
- **설명**: 코드베이스를 단순히 파일 목록이나 모듈 트리가 아니라, 횡단 관심사(인프라/프레임워크/미들웨어)와 종단 의존성(feature → 인프라/데이터/로직/인터페이스)의 두 축으로 동시에 이해할 수 있어야 한다.
- **요구사항 근거**: #4, #5, #9

### V3: 최종 사용자 관점의 표현
- **유형**: 구체적
- **설명**: 모든 feature는 개발자 용어가 아니라 최종 사용자가 무엇을 할 수 있는지(인수 테스트 형태)로 표현된다. 비개발자도 제품의 능력을 한눈에 이해할 수 있다.
- **요구사항 근거**: #2, #8

### V4: 코드와 일치하는 살아있는 문서
- **유형**: 추상적
- **설명**: 문서는 코드 변경에 뒤처지지 않는다. 실제 로직과 테스트 코드를 근거로 LLM이 문서를 갱신하므로, 문서는 코드와 함께 진화한다.
- **요구사항 근거**: #3, #8, #10

### V5: feature 단위의 의존성 가시성
- **유형**: 구체적
- **설명**: 어떤 feature 하나를 보면, 그것이 어떤 인프라·데이터 모델·아키텍처·미들웨어·로직·인터페이스에 의존하는지 종단으로 즉시 확인할 수 있어야 한다. 의존성은 별도의 구조화된 데이터로 보존된다.
- **요구사항 근거**: #9

### V6: 사용자가 통제하는 자격증명·비용
- **유형**: 구체적
- **설명**: 사용자는 GitHub App 설치를 통해 어떤 저장소에 접근을 허용할지 직접 정하고 언제든 회수할 수 있으며, 자신의 LLM API Key를 직접 등록·교체·회수할 수 있다. 호출과 비용 모두 사용자 통제 하에 있으며, 제품은 자격증명을 안전하게 보관·사용한다.
- **요구사항 근거**: #12

### V7: 모바일에서 즉시 검토·수정 가능
- **유형**: 구체적
- **설명**: 주된 사용 환경이 모바일임을 전제로, 좁은 화면과 짧은 세션에 맞춘 검토·승인·소규모 수정 흐름을 우선 제공한다. 본격적인 분석 작업은 백그라운드에서 진행되어도, 사용자는 모바일에서 결과를 빠르게 확인하고 결정할 수 있다.
- **요구사항 근거**: #13

### V8: 운영 환경에서의 안정적 가용성
- **유형**: 추상적
- **설명**: k8s 환경에서 다수 사용자/저장소를 동시에 안정적으로 처리할 수 있어야 한다. 분석은 시간이 걸리는 작업이므로 비동기·재시도·관측 가능성을 갖춰야 한다.
- **요구사항 근거**: #11

---

## 가치 간 관계 메모

- V1, V2는 분석 파이프라인(횡단 분석 → 전략 → feature 추출)의 토대.
- V3, V4는 표현 방식과 갱신 방식에 대한 약속.
- V5는 feature 문서가 단순 텍스트가 아니라 그래프/데이터로도 존재해야 한다는 요구.
- V6, V7, V8은 비기능적 토대(자격증명·UX·운영).
- V3는 V1과 V4 위에서만 의미가 있다 (잘못 발견된 feature를 사용자 관점으로 표현해도 가치 없음).
