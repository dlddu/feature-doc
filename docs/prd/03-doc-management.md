# PRD: Feature 단위 문서 관리 (LLM 활용 CRUD)

> 요구사항 #10에 대응하는 PRD.
> PRD-1, PRD-2가 자동으로 만들어주는 결과 위에서, 사용자가 LLM을 사용해 feature 문서를 손쉽게 추가/수정/삭제하는 흐름을 정의한다.

## 달성 가치
- **V3: 최종 사용자 관점의 표현** — 사용자 편집 결과도 같은 표현 형식을 따라야 한다.
- **V4: 코드와 일치하는 살아있는 문서** — 사용자 편집은 코드와 분리되지 않고 항상 코드 근거와 함께 보관된다.
- **V7: 모바일에서 즉시 검토·수정 가능** — CRUD 흐름의 UX는 모바일을 우선한다.

## 범위 / 비범위
- **범위**: feature 1개 단위로 인수 테스트 문서 / 메타데이터 / 의존성 메모를 사용자가 LLM 도움을 받아 추가·수정·삭제하는 흐름.
- **비범위**: feature 자체를 새로 발견하는 일(PRD-1), 의존성 데이터의 자동 추출(PRD-2 AC2.4).

## Acceptance Criteria

### AC3.1: feature 문서의 LLM 보조 수정
- **설명**: 사용자는 feature의 인수 테스트 문서/설명 등을 자연어 지시("이 시나리오 더 명확하게", "에러 케이스 추가") 한 줄로 LLM에게 요청해 수정할 수 있다. LLM은 변경 제안을 diff 형태로 먼저 보여주고, 사용자가 승인해야 적용된다.
- **달성 가치**: V3, V4, V7
- **검증 방법**: 모바일에서 한 손으로 "수정 요청 → diff 확인 → 승인/거부"가 3탭 이내로 완료된다. 거부된 제안은 기록되어 다음 제안 시 회피된다.

### AC3.2: feature 문서의 추가
- **설명**: 사용자는 자동 추출되지 않은 feature를 직접 추가할 수 있다. 추가 시에도 LLM이 코드베이스를 참조해 인수 테스트 초안과 의존성 후보를 제안하며, 사용자가 확정한다.
- **달성 가치**: V1, V3, V7
- **검증 방법**: 사용자가 "회원 탈퇴 기능을 추가하고 싶다"고 입력하면, LLM이 코드 검색으로 관련 위치를 찾아 초안을 제시한다. 코드에서 근거를 찾지 못하면 "근거 없음"을 명시하며 강제로 만들어내지 않는다.

### AC3.3: feature 문서의 삭제와 보존
- **설명**: 사용자는 feature 문서를 삭제할 수 있으나, 삭제는 즉시 영구 제거되지 않고 보관소(soft delete)로 옮겨진다. 사유를 함께 기록할 수 있고, 같은 feature가 다음 자동 분석에서 다시 발견되면 "이전에 거부된 항목입니다"로 표시된다.
- **달성 가치**: V4
- **검증 방법**: 삭제된 feature는 일정 기간 복구 가능. 자동 재발견 시 사용자 의도와 충돌하지 않도록 표시된다.

### AC3.4: 변경 이력 추적과 출처 보존
- **설명**: feature 문서의 모든 변경(자동 분석에 의한 변경, 사용자 LLM 보조 수정, 사용자 직접 편집)은 출처(자동/사용자 by LLM/사용자 직접)와 함께 이력으로 보존된다.
- **달성 가치**: V4
- **검증 방법**: 한 feature의 임의 시점 상태를 복원 가능. 각 변경의 출처를 모바일 화면에서 한눈에 확인 가능.

### AC3.5: 코드 자동 분석과 사용자 편집의 충돌 처리
- **설명**: 자동 재분석 결과가 사용자의 마지막 편집과 충돌할 경우, 시스템은 자동으로 덮어쓰지 않고 충돌을 사용자에게 표시한다. 사용자는 자동 결과 채택/사용자 편집 유지/병합(LLM 보조)을 선택할 수 있다.
- **달성 가치**: V4, V7
- **검증 방법**: 충돌이 발생한 feature는 명확히 표시되며, 사용자 결정 없이는 자동 결과로 덮이지 않는다.

## AC 간 관계
- AC3.1, AC3.2, AC3.3은 CRUD의 핵심.
- AC3.4는 모든 CRUD 위에 깔리는 횡단 요구.
- AC3.5는 PRD-1/PRD-2의 자동 흐름과 본 PRD의 사용자 편집 흐름 사이의 경계 보호.
