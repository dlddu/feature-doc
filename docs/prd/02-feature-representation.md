# PRD: Feature 표현 (인수 테스트 + 종단 의존성)

> 요구사항 #8, #9를 묶은 PRD.
> PRD-1에서 추출된 feature 후보가 확정된 이후, 각 feature를 어떤 형태로 표현하고 어떤 데이터를 함께 보존하는지를 정의한다.

## 달성 가치
- **V3: 최종 사용자 관점의 표현** — 인수 테스트 형식이 핵심.
- **V4: 코드와 일치하는 살아있는 문서** — 표현의 근거는 항상 실제 로직 + 테스트 코드.
- **V5: feature 단위의 의존성 가시성** — 종단 의존성이 별도 데이터로 보존되어야 한다.

## 범위 / 비범위
- **범위**: 확정된 feature 1개에 대한 (a) 인수 테스트 표현 생성, (b) 종단 의존성 추출/저장/조회.
- **비범위**: feature 후보 추출(PRD-1), 사용자가 수동으로 인수 테스트나 의존성을 편집하는 흐름(PRD-3).

## Acceptance Criteria

### AC2.1: 로직 코드로부터 인수 기준 도출
- **설명**: 시스템은 feature가 가리키는 코드(엔드포인트 핸들러, UI 컴포넌트, CLI 명령 등)의 실제 로직을 LLM으로 분석하여, 최종 사용자가 그것을 사용했을 때 일어나는 일을 인수 기준 형태("주어진 ~ / ~ 했을 때 / ~ 해야 한다")로 표현한다.
- **달성 가치**: V3, V4
- **검증 방법**: 한 feature에 대해 생성된 인수 기준이 실제 로직의 분기/사이드 이펙트와 일치하는지 사용자가 검토 가능. 각 인수 기준에는 근거가 된 코드 위치가 첨부된다.

### AC2.2: 테스트 코드로부터 인수 기준 보강
- **설명**: 시스템은 동일 feature와 연관된 테스트 코드(단위/통합/E2E)를 추가로 분석하여, AC2.1의 인수 기준을 보강하거나 누락된 시나리오(에러 케이스, 경계 조건)를 추가한다. 로직 분석과 테스트 분석에서 모순이 발견되면 사용자에게 표시한다.
- **달성 가치**: V3, V4
- **검증 방법**: 테스트가 존재하는 feature는 인수 기준의 시나리오 수가 로직만 본 결과보다 같거나 많다. 모순 케이스는 별도 섹션("코드와 테스트가 다르게 말하는 부분")으로 노출된다.

### AC2.3: 최종 사용자 관점의 인수 테스트 문서 생성
- **설명**: AC2.1, AC2.2의 결과를 합쳐 feature 1개당 1개의 인수 테스트 문서를 생성한다. 문서는 개발자 용어가 아니라 최종 사용자가 알아볼 수 있는 언어로 작성되며, 시나리오 단위로 구조화된다.
- **달성 가치**: V3
- **검증 방법**: 비개발자(예: 제품 소유자)가 문서를 읽고 그 feature가 무엇을 하는지 설명할 수 있다. 모든 시나리오에 근거 코드/테스트 위치가 링크된다.

### AC2.4: feature 단위 종단 의존성 추출
- **설명**: 시스템은 feature 1개에 대해, 그것이 의존하는 인프라(외부 서비스/큐/DB), 데이터 모델(엔티티/스키마), 아키텍처 레이어, 프레임워크/라이브러리, 미들웨어, 핵심 로직 모듈, 외부 인터페이스(API/UI/CLI 진입점)를 종단으로 식별한다.
- **달성 가치**: V5, V2
- **검증 방법**: feature를 변경할 때 어떤 컴포넌트가 영향받는지 의존성 목록만 봐도 추론 가능. 각 의존성에는 코드 근거가 첨부된다.

### AC2.5: 의존성을 별도 구조화 데이터로 저장
- **설명**: 추출된 의존성은 단순 텍스트 문서가 아니라, feature ID를 키로 하는 구조화된 데이터(예: feature ↔ 인프라/데이터/모듈 간 다대다 관계 그래프)로 저장된다. 다른 feature의 의존성과 비교·교차 조회가 가능해야 한다.
- **달성 가치**: V5
- **검증 방법**: "데이터 모델 X를 사용하는 feature 전부", "미들웨어 Y에 영향받는 feature 전부"와 같은 역방향 질의가 가능. 데이터는 시스템 외부로 export 가능.

### AC2.6: 코드 변경 시 표현·의존성의 재계산 가능성
- **설명**: 코드가 변경되어 재분석을 트리거하면, 영향받는 feature의 인수 테스트 문서와 의존성 데이터가 갱신된다. 사용자에게는 이전 버전과의 차이가 표시된다.
- **달성 가치**: V4, V5
- **검증 방법**: 동일 feature를 두 시점에 분석하면 변경된 시나리오/추가·제거된 의존성이 diff 형태로 노출된다.

## AC 간 관계
- AC2.1 → AC2.2 → AC2.3 의 순서로 인수 테스트 표현이 완성된다.
- AC2.4 → AC2.5 는 의존성 추출과 저장의 두 단계.
- AC2.6은 (AC2.3, AC2.5)의 재실행 가능성을 보장하는 요구.
